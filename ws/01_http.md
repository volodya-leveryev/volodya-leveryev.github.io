## Протокол HTTP

**Семиуровневая модель OSI:**
- 7 уровень (высший), прикладной. На этом уровне работают протоколы с которыми работают прикладные приложения (HTTP, SMTP, POP3, IMAP и др.).
- 6 уровень, презентационный (представление информации). На этом уровне решаются вопросы кодирования данных (в какой кодировке их передавать), а также шифрования и сжатия данных. Часть функций этого уровня берут на себя прикладные приложения.
- 5 уровень, сеансовый. Прикладные приложения обычно выполняют функции управления сеансами связи самостоятельно и пропускают этот уровень.
- 4 уровень, транспортный. На этом уровне происходит распределение передаваемых данных по приложениям с помощью портов TCP или UDP. Каждый порт соответствует максимум одному процессу (точнее сокету), один процесс может занять несколько портов.
- 3 уровень, сетевой. На этом уровне данные должны быть переданы от одного компьютера другому. При необходимости с помощью сетевых маршрутизаторов происходит передача данных между сетями. Основной протокол этого уровня — IP. На этом уровне каждой сетевой карте компьютера назначается как минимум один IP-адрес. Один компьютер может иметь несколько IP-адресов.
- 2 уровень, канальный. На этом уровне происходит работа локальных компьютерных сетей (Ethernet, Wi-Fi, и др.). Также на этом уровне каждой сетевой карте назначается MAC-адрес.
- 1 уровень (низший), физический. На этом уровне протоколы работы локальных компьютерных сетей описывают как физически происходит передача данных между абонентами (например, как физические сигналы кодируют биты данных).

Большинство прикладных протоколов полагаются на TCP, который на транспортном уровне реализует не только распределение по приложениям с помощью портов, но создает сетевые соединения. При этом данные из разных пакетов (TCP-сегментов) объединяются в потоки последовательных байтов. TCP также проверяет дошли ли данные данные до получателя и при необходимости пересылает их заново. К сожалению, для этого необходимо некоторое количество накладных расходов в виде передачи дополнительных служебных пакетов. Например, перед открытием каждого TCP-соединения клиенту и серверу нужно обменяться 3 служебными пакетами.

Обратите внимание, что UDP не создает потоков и передает данные порциями (датаграммами), то есть он не работает на этом уровне.

- большие задержки при открытии соединения, при проверке доставки данных.

TCP-соединение идентифицируется:
- IP-адрес клиента
- TCP-порт клиентского приложения (16-битный)
- IP-адрес сервера
- TCP-порт серверного приложения (16-битный)


На этому уровне работает протокол SSL/TLS.

**Версии протокола HTTP:**
- 0.9 — больше не используется.
- 1.0 — практически не используется.
- 1.1 — активно применяется.
- 2 — активно применяется, дает улучшения по задержкам за счет повторного использования однажды открытых TCP-соединений.
- 3 — находится в стадии экспериментального использования. Вместо TCP использует UDP и осуществляет сборку данных самостоятельно. Улучшает производительность передачи данных за счёт экономии на накладных расходах протокола TCP.

HTTP использует TCP-порт 80
HTTPS использует TCP-порт 443

Клиент отправляя запросы указывает адрес ресурса на сервере (path).

URL ­— Uniform Resource Locator:
схема (https)
имя пользователя и пароль (user@, user:1234@)
доменное имя сервера (www.google.com)
номер TCP-порта (:8000)
путь к ресурсу (/index.html)
идентификатор фрагмента страницы, anchor (#tagid)
параметры запроса (?param1=value&param2=value)

Формат запроса (request)
GET /pub/WWW/TheProject.html HTTP/1.1
Accept-Encoding: gzip, deflate, br
Host: www.tutorialspoint.com

Формат ответа (response)
HTTP/1.1 200 OK
Date: Mon, 27 Jul 2009 12:28:53 GMT
Server: Apache/2.2.14 (Win32)
Last-Modified: Wed, 22 Jul 2009 19:15:56 GMT
Content-Length: 88
Content-Type: text/html
Connection: Closed

<html>
<body>
<h1>Hello, World!</h1>
</body>
</html>

Статусы
1хх — информация
2хх — успех
3хх — перенаправление
4хх — ошибка со стороны клиента
5хх — ошибка со стороны сервера

Методы
GET — получить ресурс
HEAD — получить заголовки от ресурса (то же что и GET только без тела ответа)
OPTIONS — получить опции для ресурса
POST — отправить данные на обработку
PUT — создать ресурс 
PATCH — исправить ресурс
DELETE — удалить ресурс
TRACE — для отладки
CONNECT — для создания HTTPS-соединения

Самые популярные заголовки — передают дополнительную информацию о запросе и ответе.
Host — определяет целевой веб-сайт
User-Agent — информация о клиенте
Accept — допустимые MIME-типы ответов
Content-Type — MIME-тип ответа
Accept-Language — языки пользователя
Accept-Encoding — способ кодирования данных

MIME-тип — это строка, которая идентифицирует тип контента:
text/plain — просто текст
text/html — HTML-страница
image/png — изображение в формате PNG

HTTP — изначально stateless протокол, т.е. он не хранит состояние. Это позволяет упростить разработку веб-сервера и упростить масштабирование веб-сервера.

Куки — текстовая строка которая идентифицирует специфичные для пользователя данные (например идентификатор сессии)
Set-Cookie ­— заголовок ответа, который устанавливает Cookie
Cookie — заголовок запроса, который содержит текущую установленную Cookie

Благодаря механизму Cookie в HTTP появляется поддержка состояния (State).
