# Документирование API в формате Swagger / OpenAPI
## Введение

В монолитной архитектуре модули общаются друг с другом через вызовы функций. В микросервисной архитектуре система состоит из множества независимых сервисов, которые общаются по сети (чаще всего через REST API по HTTP). В таких условиях возникает проблема: как команде, разрабатывающей микросервис `A`, узнать, какие данные ожидает и возвращает микросервис `B`? Лучший API бесполезен, если никто не знает, как им пользоваться. Здесь на помощь приходит стандартизированное документирование API.

## 1. OpenAPI vs Swagger: В чем разница?

Частая путаница возникает между терминами Swagger и OpenAPI. На собеседованиях это популярный вопрос.

- **Swagger 2.0** — это формат описания RESTful API, созданный компанией SmartBear.
- **OpenAPI Specification (OAS)** — это открытый стандарт разработки интерфейсов API. В 2015 году SmartBear передала Swagger под управление Linux Foundation (OpenAPI Initiative). С версии 3.0 стандарт стал называться OpenAPI.
- **Современное правило: OpenAPI** — это сама спецификация (правила написания документа), а **Swagger** — это набор инструментов (экосистема) для работы с этой спецификацией.

## 2. Анатомия спецификации OpenAPI

OpenAPI-документ пишется в формате **YAML** или **JSON**. YAML предпочтительнее для людей, так как он легче читается.

Современный стандарт (OAS 3.0 / 3.1) состоит из нескольких ключевых блоков:

1. **openapi**: Версия спецификации (например, `3.0.3`).
2. **info**: Метаданные об API (название, описание, версия, контакты).
3. **servers**: Базовые URL, по которым доступен API (например, `dev`, `stage`, `prod`).
4. **paths (Маршруты)**: Самая важная часть. Описание всех эндпоинтов (`/users`, `/orders`), доступных HTTP-методов (`GET`, `POST`), их параметров, тела запроса и возможных ответов (включая коды ошибок).
5. **components**: Переиспользуемые объекты. Здесь описываются схемы данных (DTO), параметры, ответы и схемы безопасности (например, JWT или OAuth2), на которые можно ссылаться из блока `paths` с помощью директивы `$ref`.

## Пример OpenAPI спецификации

**openapi.yaml**
```yaml 
openapi: 3.0.3
info:
  title: User Service API
  description: Микросервис для управления пользователями
  version: 1.0.0
servers:
  - url: https://api.example.com/v1
paths:
  /users/{id}:
    get:
      summary: Получить пользователя по ID
      parameters:
        - name: id
          in: path
          required: true
          description: Уникальный идентификатор пользователя
          schema:
            type: integer
            format: int64
      responses:
        '200':
          description: Успешный ответ
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '404':
          description: Пользователь не найден
components:
  schemas:
    User:
      type: object
      properties:
        id:
          type: integer
          example: 123
        username:
          type: string
          example: "ivan_ivanov"
        email:
          type: string
          example: "ivan@example.com"
```

## 3. Инфраструктура Swagger (Инструменты)
Как мы уже выяснили, Swagger — это инструментарий. Вот основные из них:

- **Swagger Editor**: Браузерный редактор (часто с разделением экрана: слева YAML, справа визуализация), который валидирует код в реальном времени.
- **Swagger UI**: Инструмент, который превращает YAML/JSON файл в красивую, интерактивную веб-страницу. Разработчики могут не только читать документацию, но и отправлять реальные запросы к API прямо из браузера (кнопка *Try it out*).
- **Swagger Codegen / OpenAPI Generator**: Мощная утилита, которая позволяет на основе файла OpenAPI сгенерировать:
  * Каркас сервера (серверные стабы на Spring Boot, Node.js, Python и др.).
  * Клиентские SDK (готовые классы для фронтенда на TypeScript, либо для других микросервисов на Java/C#).

## 4. Подходы к созданию документации

Существует два глобальных подхода к работе с OpenAPI в разработке:

**Подход 1: Code-First (Сначала код)**

Разработчик пишет код микросервиса, расставляет специальные аннотации (например, `@Operation`, `@Schema` в Java/Spring-doc или декораторы в NestJS), а генератор автоматически собирает OpenAPI-файл при запуске.

- **Плюсы**: Документация всегда синхронизирована с кодом. Быстрый старт.
- **Минусы**: Клиенты (фронтенд или другие команды) должны ждать, пока вы напишете код, чтобы получить интерфейс. Раздувает код аннотациями.

**Подход 2: Design-First или API-First (Сначала проектирование)**

Команда собирается вместе и *сначала* пишет YAML-файл спецификации (Контракт). Только после согласования этого файла начинается программирование.

- **Плюсы**:
  * Фронтенд и бэкенд разработчики могут работать параллельно: фронтенд использует мок-сервер из YAML, бэкенд пишет логику.
  * API получается более продуманным, так как проектируется без привязки к конкретному языку.
- **Минусы**: Требует жесткой дисциплины, чтобы код не разошелся со спецификацией.

Для микросервисов настоятельно рекомендуется подход **Design-First/API-First**.

## 5. Роль OpenAPI в микросервисной архитектуре

Зачем нужны эти спецификации, если у нас десятки микросервисов?

1. **Контракты между сервисами**: OpenAPI выступает юридическим договором между сервисом-провайдером и сервисом-потребителем. Если микросервис «Корзина» вызывает микросервис «Каталог товаров», команда Корзины генерирует HTTP-клиент (например, FeignClient) прямо из спецификации Каталога.
2. **API Gateway**: В микросервисной архитектуре точка входа для клиентов — это API Gateway (шлюз). Многие современные шлюзы (Kong, AWS API Gateway, KrakenD) умеют импортировать OpenAPI спецификации и автоматически настраивать маршрутизацию, защиту (Rate Limiting) и валидацию входящих запросов по схеме DTO еще до того, как запрос дойдет до самого микросервиса.
3. **Контрактное тестирование (Contract Testing)**: Спецификации OpenAPI используются для автоматизированного тестирования. Инструменты проверяют, не нарушил ли сервис свой заявленный контракт после деплоя новой версии (чтобы не сломать соседние микросервисы).
4. **Единый портал разработчика (Developer Portal)**: Инструменты вроде Backstage от Spotify собирают OpenAPI файлы со всех 50-100 микросервисов компании и создают единый внутренний каталог всех доступных API в компании.

## Заключение

Документирование API — это не просто прихоть, а критически важный процесс в микросервисной разработке. Спецификация OpenAPI является стандартом де-факто для REST API. Она не только помогает писать документацию, которую могут читать люди (через Swagger UI), но и предоставляет машиночитаемое описание вашего сервиса, позволяя автоматизировать генерацию кода, валидацию запросов и тестирование.
